## Status honesty (read first)
**Wan-Streamer v0.1 is REAL and research-stage.** Verified: arXiv 2606.25041, "Wan-Streamer v0.1:
End-to-end Real-time Interactive Foundation Models," Wan Team (Alibaba). A **single unified end-to-end
Transformer** with **block-causal attention**, all modalities (language/audio/video) as input+output
tokens in one **temporally-causal sequence**; explicitly rejects the cascaded VAD→ASR→TTS→avatar
pipeline; **full-duplex** (perceives while generating); reported **~200 ms model-side / ~550 ms total**
latency. NOT verified / unknown as of this writing: public **API**, **open weights** for the streamer
variant, **GPU footprint**, third-party reproduction. Treat performance numbers as author-reported and
**pending independent validation.** ⇒ Design for it now; **do not depend on it** for 1.0.

## The one architectural decision that matters
Do NOT hard-wire Wan-Streamer. Make **Avatar Runtime a provider behind a Render Provider Router with a
stable `AvatarProvider` interface** — exactly the `RenderProviderAdapter` pattern we already use for
Grok. Wan-Streamer becomes ONE future provider; today's providers (GPT-4o Realtime, Gemini Live,
HeyGen, Tavus, HyperFrames, LivePortrait) plug into the same interface. This gives swap-without-rewrite
and lets 1.0 ship on mature providers while Wan-Streamer matures.

## 1. Wan-Streamer architecture — what to reuse vs replace
Component breakdown (verified where cited; else [inferred] from the unified-model class):
| Block | What it does | For EPIC💀GRAM |
|---|---|---|
| **Video Tokenizer** [inferred] | encodes/decodes video frames ↔ discrete/continuous tokens in the causal stream | **use as-is** (it IS the model); we don't rebuild it |
| **Audio Encoder** [inferred] | streaming audio → tokens (no separate ASR) | **use as-is** |
| **Vision Encoder** [inferred] | camera/screen frames → perception tokens | **use as-is for the avatar's own perception**, but EPIC keeps its OWN Perception OS (screen/browser/geo) upstream — don't route all platform perception through the avatar model |
| **Unified Transformer** (verified) | block-causal attention over one temporally-causal multimodal sequence | **the core; use as-is** — this is the value |
| **Streaming Decoder** [inferred] | emits audio+video tokens incrementally | **use as-is**; wrap with our frame/audio queues |
| **Real-Time Rendering** [inferred] | tokens → displayable video/audio | **use**, but our client handles WebRTC transport |
| **Full-Duplex** (verified) | keep listening while speaking | **use** — this is what modular stacks can't do |
| **Latency pipeline** (verified ~200/550 ms) | end-to-end causal, no step boundaries | **use**; validate on our GPU |

**Replace/keep external (NOT Wan-Streamer's job):** long-term **Memory**, **Planner/Reasoning** at the
operator level, **multi-agent orchestration**, **Telegram/Browser/Computer-Use actions**, **content
pipeline**, **business logic**. Wan-Streamer is a **perception+dialogue+face/voice generation runtime**,
not the operator. It closes the Avatar Runtime box; it does not replace EPIC💀GRAM OS.

## 2. Ideal architecture
```
User
 └─ Camera ─┐
 └─ Mic ────┤
 └─ Screen ─┘
        ▼
  AI Perception OS  (vision/audio/screen/browser/geo — EPIC-owned adapters)
        ▼
  World Model  ←→  Memory System (long-term, single authority)
        ▼
  Planner → Reasoning  (AI Operator Runtime; per-role model router)
        ▼
  ┌─────────────  Avatar Runtime (provider behind router)  ─────────────┐
  │   Wan-Streamer (future)  |  GPT-4o Realtime / Gemini Live / HeyGen / │
  │   Tavus / HyperFrames / LivePortrait (now)                           │
  │   → generates: Video (face) · Voice · Emotion/expression  (synced)   │
  └─────────────────────────────────────────────────────────────────────┘
        ▼
  Action Layer:  Computer Use · Browser Use · Telegram Runtime · Social/Publish
        ▼
  World  (all side-effects MANUAL_APPROVAL_ONLY)
```
Key: the avatar model's OWN perception (its vision/audio encoders) is used for **conversational
grounding**; EPIC's **Perception OS + World Model + Memory** remain the platform's source of truth and
feed context in. Two perception layers, one authority (the platform).

## 3. Data flow
- **Video in:** camera/screen → Perception OS (consent-gated) → (a) World Model, (b) avatar provider as
  visual context tokens.
- **Voice in:** mic → provider audio-in (streaming). Full-duplex: NOT gated behind end-of-turn VAD.
- **Emotion:** two sources — (a) generated by the avatar model as part of the video stream (mouth,
  brows, gaze); (b) optional high-level affect target from the Planner ("empathetic", "excited") pushed
  via `pushEmotion()` as a conditioning signal. Never a separate lip-sync stage.
- **Memory:** Planner retrieves from Memory Authority → injected as context/system tokens per turn;
  observations written back through the single writer (never by N runtimes).
- **Agent actions:** Reasoning decides an action → Action Layer (Computer/Browser/Telegram) executes →
  result returns to World Model → next turn. Actions stay approval-gated.
- **Speech + face generation:** emitted TOGETHER by the unified model (that's the point) → Streaming
  Decoder → frame queue + audio queue → WebRTC to client, A/V synced.
- **Reaction to user:** because full-duplex, an incoming barge-in updates the input stream mid-generation
  → provider `interrupt()` → graceful turn-take.

## 4. Real-time architecture (feel-alive)
```
GPU node (avatar provider) ──tokens──► Streaming Decoder
   ├─ Video Queue ──┐
   ├─ Audio Queue ──┼─ A/V sync (PTS/clock) ─► WebRTC (LiveKit SFU) ─► client <video>
   └─ Emotion/meta ─┘
Client mic/cam ─► WebRTC ─► ingest ─► provider input stream (full-duplex)
```
- **Transport:** WebRTC via **LiveKit** (SFU) for sub-second bidirectional A/V; Realtime/Streaming APIs
  where a provider offers them (GPT-4o Realtime, Gemini Live).
- **Queues + sync:** separate **frame queue** and **audio queue**, reconciled on a shared clock (PTS) so
  lips match voice; drop-late policy on video, never on audio.
- **Low latency levers:** causal/streaming model (no full-utterance wait), chunked token emit, jitter
  buffer tuned ~60–120 ms, GPU co-located with SFU region, warm sessions (pre-created `createSession`).
- **"Alive" = ** <~600 ms round-trip + full-duplex barge-in + continuous micro-motion (blink, gaze,
  breath) even when idle. Wan-Streamer's ~550 ms total is in this window **if** it reproduces on our GPU.

= Seedance/Kling/Hunyuan/Flow/OmniHuman. Different jobs — both live behind the same router, different
capability flags.

## 6. Modular architecture — Avatar/Render Provider Router
Extend the existing `RenderProviderAdapter` pattern into an `AvatarProvider` registry:
- Router selects provider by **capability flags** (`realtime`, `full_duplex`, `face`, `voice`,
  `perception`, `offline_clip`) + cost + region + availability.
- Live conversation → realtime provider; content clip → video-gen provider. Same interface, different
  flags. Wan-Streamer registers with all flags true (when available); HeyGen with `face+voice` but not
  `full_duplex`; Seedance with `offline_clip` only.
- Fallback chains (e.g., Wan-Streamer → Tavus → GPT-4o+HeyGen) so a provider outage degrades gracefully.

## 7. API interface (stable contract)
```ts
interface AvatarProvider {
  id: string; capabilities: AvatarCap[];              // realtime|full_duplex|face|voice|perception|offline_clip
  createSession(cfg: AvatarSessionCfg): Promise<SessionHandle>;  // warm the GPU/session
  pushAudio(s, pcmChunk): void;      // streaming mic in (full-duplex; no end-of-turn gate)
  pushVideo(s, frame): void;         // camera in (conversational grounding)
  pushFrame(s, screenFrame): void;   // screen/context frame in
  pushEmotion(s, affect): void;      // high-level affect target (conditioning, not lip-sync)
  render(s): AsyncStream<AvOut>;      // out: synced {videoFrame, audioChunk, meta} stream
  interrupt(s): void;                 // barge-in / turn-take
  stop(s): void;                      // end current generation
  close(s): Promise<void>;            // tear down session
}
```
Non-realtime providers (Seedance/HeyGen batch) implement a `renderClip(prompt|refs) → assetUrl` variant
on the same registry; the router routes by capability. All outputs still enter Quality Gate → Approval;
no auto-publish.

## 8. Roadmap (v2.0; each phase approval-gated, MANUAL_APPROVAL_ONLY)
- **Phase 1 — MVP (now/near):** finish EPIC💀GRAM 1.0; close **P30.1 real Grok asset** (still open);
  ship the manual create→scene→media→approve loop. *Dep:* real provider path. *Risk:* scope creep.
  *Acceptance:* real Grok asset pending_review in Candidate Groups.
- **Phase 2 — Realtime Avatar:** `AvatarProvider` interface + router + LiveKit transport; first live
  provider = Tavus OR GPT-4o Realtime + HeyGen face. *Dep:* LiveKit, provider accounts. *Risk:* latency,
  cost. *Acceptance:* <1 s round-trip live talking avatar, human-approved to go live.
- **Phase 3 — Emotional Avatar:** `pushEmotion` conditioning + affect from Planner; continuous
  micro-motion; barge-in via `interrupt`. *Dep:* Phase 2. *Risk:* uncanny valley. *Acceptance:*
  full-duplex turn-taking + expression that reads as natural in blind review.
- **Phase 4 — Autonomous AI Operator:** Avatar Runtime wired to Perception OS + Memory + Action Layer;
  supervised autonomy (draft→approve). *Dep:* Perception OS, Memory Authority, Action Layer. *Risk:*
  safety/impersonation. *Acceptance:* operator perceives→reasons→acts with human approval on every
  side-effect.
- **Phase 5 — Multi-Agent Live Ecosystem:** Agent Mesh of avatar operators (Cast), one Director;
  cross-character live interaction. *Dep:* Identity Graph, Agent Mesh. *Risk:* orchestration complexity,
  cost, moderation. *Acceptance:* ≥2 live avatars interacting under supervision, no autonomous publish.
- **Wan-Streamer insertion point:** slots into Phase 2/3 as a provider **only once** it has usable
  weights/API + reproduces latency on our GPU. Until then Phases 2–3 run on Tavus/GPT-4o+HeyGen; the
  interface makes the swap a config change.

## 9. Wan-Streamer as NOVIKOVA's core — assessment
**Conceptually the best long-term core:** a single model that sees+hears+reasons+speaks+shows a face
full-duplex is exactly NOVIKOVA's end state (living operator, not chatbot). **But do NOT make it the
core now.** Fundamental blockers today:
- **Research status (v0.1):** no confirmed public API/open weights for the streamer variant; expect
  churn and no SLA.
- **Compute:** unified realtime audio-video generation is GPU-heavy; continuous interactive operation
  (a persona that's "always live") multiplies cost — needs real GPU budget + autoscaling.
- **Unverified numbers:** ~200/550 ms is author-reported; must be reproduced on our hardware.
- **Not an operator:** it won't do Memory, Telegram, Browser, orchestration, business logic — those stay
  EPIC-owned regardless.
**Practical path for NOVIKOVA now:** identity/voice/looks via our Passport (Identity + Voice + Wardrobe
packs); live presence via **Tavus** or **GPT-4o Realtime + HeyGen** behind the `AvatarProvider` router;
content clips via Grok/Seedance/Kling. When Wan-Streamer ships usable weights/API and passes our latency
+ cost validation, register it as a provider and A/B it against the incumbent — **swap, don't rebuild.**
